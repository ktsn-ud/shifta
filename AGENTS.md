# AGENTS.md

## 目的

このリポジトリで作業するエージェント（人間含む）が、同じ前提と品質基準で開発できるようにする。

## プロジェクト概要

- 個人用のシフト管理・概算給与管理Webアプリ。
- 複数勤務先、給与ルール履歴、塾の時間割シフトを扱う。
- Google Calendar と同期するが、データの正はアプリDB。

## 開発原則

- データの正はアプリケーションDB。Google Calendar は表示・補助用途。
- Google Calendar 側で編集された内容はアプリへ逆同期しない。
- MVP範囲を優先し、未定義の拡張要件（控除・交通費・複数ユーザー等）は先に追加しない。
- 設計責任は人間が持ち、エージェントは実装単位で支援する。

## 現在の技術スタック

- Next.js 16 (App Router)
- React 19
- TypeScript (strict)
- Tailwind CSS 4
- ESLint 9 + eslint-config-next
- Prettier

## 実行コマンド

- 開発: `pnpm dev`
- ビルド: `pnpm build`
- 本番起動: `pnpm start`
- 型チェック: `pnpm exec tsc --noEmit`
- Lint: `pnpm lint`
- Format: `pnpm format`

## MCP活用方針

- `next-devtools` を積極利用し、Next.js関連タスクでは実装前にランタイム情報を確認する。
- Next.jsの調査・実装時は、まず `next-devtools` の `init` と `nextjs_index` を実行し、利用可能ツールとサーバー状態を把握する。
- ルート・エラー・ビルド状態の確認は `nextjs_call` を優先し、必要時のみ補助的に他手段を使う。
- UIコンポーネント追加・調査は `shadcn` MCP を優先し、レジストリ検索・実装例取得・追加コマンド生成を利用する。
- MCPが利用不可または情報不足の場合のみ、ローカル調査や通常コマンドにフォールバックする。

## ディレクトリ方針

- UIルート: `app/`
- 将来の想定機能ルート: `app/calendar`, `app/summary`, `app/workplaces`, `app/shifts`
- 共通コンポーネント: `components/`
- 業務ロジック: `lib/`（`lib/payroll`, `lib/calendar-sync`, `lib/auth` など）
- DBスキーマ: `prisma/schema.prisma`
- 設計ドキュメント: `docs/`

## 実装ガイドライン

- タスク着手前に `docs/DESIGN_SPECIFICATION.md` で要件・仕様を確認する。
- TypeScript の `any` は原則使わない。型安全を優先する。
- 画面ロジックと業務ロジックを分離し、計算処理は `lib/` に置く。
- import は `@/*` エイリアスを優先して可読性を保つ。
- 変更は最小差分で行い、無関係なリファクタは混ぜない。

## 設計ドキュメント参照ガイド

`docs/DESIGN_SPECIFICATION.md` は以下のセクションで構成されている。実装時の参考セクション：

| タスク種別                 | 参照セクション                       | 内容                                            |
| -------------------------- | ------------------------------------ | ----------------------------------------------- |
| **画面実装**               | 2.1（ルーティング）+ 9.3（画面仕様） | URL構造と画面定義、フォーム仕様、バリデーション |
| **API/DB操作**             | 3（ドメインモデル）                  | Entity定義、関連性、制約条件                    |
| **給与計算機能**           | 8（給与計算ロジック）                | 計算式、深夜・残業・休日の考慮                  |
| **Google Calendar同期**    | 7（外部連携）                        | 同期フロー、イベント属性、失敗処理              |
| **ユースケース検証**       | 6（ユースケース）                    | 業務フロー、登録・編集・集計の流れ              |
| **テスト・バリデーション** | 10（バリデーション・エラー処理）     | クライアント・サーバー側のエラーハンドリング    |

**参照例**

- SCR_004 (シフト入力) を実装する場合：
  1. 2.1 でパス `/my/shifts/new` の構造を確認
  2. 3.2, 3.3, 3.5 でWorkplace, PayrollRule, Shift の仕様確認
  3. 9.3 SCR_004 にて フォーム定義、バリデーション、エラーメッセージ確認
  4. 7.2 Google Calendar同期フロー確認
  5. 10章 バリデーション・エラー処理確認

---

1. **設計書確認**: `docs/DESIGN_SPECIFICATION.md` で以下を確認
   - 実装対象の画面 ID（SCR\_###）と仕様
   - 影響するドメインモデル（3章）
   - 関連する給与計算やビジネスロジック（8章）
   - バリデーション・エラー仕様（10章）
2. **影響範囲特定**: UI / API / DB スキーマ / 外部連携のいずれが対象か判定
3. **実装**: ガイドラインに従い、最小差分で実装
   - Prisma系コマンドのうち、エージェントが実行してよいのは `pnpm prisma generate` のみ
   - `pnpm prisma generate` が必要な場合はエージェントが実行する。ただし、エージェント環境とユーザー環境は異なるため、実装を一時停止してユーザーにも `pnpm prisma generate` の実行を依頼し、返答後に続行する
   - migrationファイルの作成はユーザーが行う。`pnpm prisma migrate dev` が必要な場合は実装を一時停止してユーザーに実行を依頼し、返答後に続行する
   - エージェントは `pnpm prisma generate` 以外の Prisma系コマンドを実行しない
   - エージェント環境とユーザー環境では `node_modules` が独立しているため、Codex起動直後の作業開始時にもエージェント側で `pnpm install` を実行して依存関係を同期する
   - `pnpm add` / `pnpm remove` / `pnpm update` など依存関係を変更するパッケージ操作は基本的にユーザーが行う。必要になった場合は実装を一時停止してユーザーに実行を依頼し、返答後に続行する
   - ユーザーがパッケージ操作を実行した後にも、エージェント側で `pnpm install` を実行して依存関係を同期する
   - もし `pnpm dev` の実行が必要な場合もエージェントは実行せず、ユーザーに依頼
4. **タスク単位の検証**: コードベースに触れる実装では、各タスクの実装後かつコミット前に `pnpm exec tsc --noEmit`, `pnpm lint`, `pnpm test`, `pnpm format` を必ず実行
5. **コミット**: 各タスクの検証・整形が通ったら、次のタスクへ進む前に Gitコミット規約に従ってコミット

## 完了条件

- 要件を満たし、既存挙動を壊していない。
- TypeScript の型エラーがない（`pnpm exec tsc --noEmit` が通る）。
- Lintエラーがない。
- 仕様とのズレがある場合、差分理由を明記して報告する。

## Gitコミットメッセージ規約

- 件名は `feat: 〇〇` の形式で記載する（`fix`, `refactor`, `docs` なども同形式）。
- `IMPLEMENTATION_TASKS.md` に基づく実装タスクのコミットでは、件名にタスク番号を含める（例: `feat: T5-2 〇〇`）。
- コミットメッセージは原則日本語で記載する。
- 件名では「何を実装/変更したか」と「なぜ必要か」が分かるように要約する。
- 件名の後は1行空け、詳細を箇条書きで記載する。
- 箇条書きには実装内容と意図を簡潔に書く。

例:

```text
feat: シフト登録フォームに勤務先絞り込みを追加

- 勤務先選択時に候補を絞り込むUIを追加
- 入力ミスを減らし、登録時間を短縮することが目的
- 既存の登録API仕様は変更せず後方互換性を維持
```

## Gitコミット粒度規約

- 1コミットは「1目的 + 1検証単位」とする。
- 件名1文で「何を・なぜ」を説明できる範囲に分割する。
- 機能追加・リファクタ・整形のみの変更は別コミットに分ける。
- 依存関係更新は単独コミットにする（`package.json` と `pnpm-lock.yaml` は同一コミット）。
- 仕様変更と、それに必要なテスト更新は同一コミットに含める。
- `revert` しても他機能を巻き込まない単位を維持する。

## 実装依頼時のコミット運用

- 「実装をお願い」と依頼された場合、まず作業を適切な粒度のタスクへ分割する。
- 作業は `実装 → 検証と整形 → コミット → 次の実装` の順で進め、すべてをまとめて実装した後に後追いで分割コミットしない。
- 各タスクのコミット前に必ず `pnpm exec tsc --noEmit`, `pnpm lint`, `pnpm test`, `pnpm format` を実行し、通過後に差分を確認してからコミットする。
- ドキュメント作成・編集のみで実際のコードベースに触れないタスクは、型チェック・Lint・テストを省略してよい。その場合も `pnpm format` を実行し、整形後の差分を確認してからコミットする。
- ただし、ユーザーから明示的に指示がない限り push は行わない。
- いかなる場合も、エージェント判断でリモートへの push は絶対に行わない。

## 禁止・注意事項

- 既存の未関連変更を巻き戻さない。
- 破壊的操作（データ削除・履歴改変）を無断で行わない。
- 仕様が曖昧な場合は推測で確定せず、前提を明記して進める。
- 判断に迷う点がある場合、独自判断で実装せず必ずユーザーに確認してから進める。
