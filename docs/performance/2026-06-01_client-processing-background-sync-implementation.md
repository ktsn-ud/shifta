# クライアント処理主導 + バックグラウンド同期 実装メモ（2026-06-01）

## 1. 目的

- 画面遷移を同期完了待ちで止めず、DB更新後すぐにユーザー操作を継続できるようにする。
- Google Calendar 同期はバックグラウンドで実行し、結果は非ブロッキング通知で扱う。
- データの正をDBに固定し、セキュリティを維持したまま体感性能を改善する。

## 2. 実装内容

- シフト系更新API（作成/更新/削除/確定/一括登録）のレスポンスに `sync` オブジェクトを追加。
- `sync` は `status`, `pending`, `ok`, `errorCode`, `requiresCalendarSetup`, `requiresSignOut` を返す。
- クライアント共通パーサを `sync` と旧 `syncStatus` 両対応に拡張。
- 成功トーストで `sync.pending=true` の場合に「バックグラウンド同期中」を明示。

## 3. セキュリティ方針

- Google API はサーバー側のみ実行し、クライアントから直接呼び出さない。
- 認証/認可は既存の `requireCurrentUser`・所有権検証・CSRF検証を維持。
- クライアントへの返却は同期状態に必要な最小情報に限定する。

## 4. 今回の非対象

- 外部ジョブキュー基盤の導入
- Google Calendar 逆同期
- 同期失敗の自動再試行ワーカー常駐化

## 5. 追加実装（同日フェーズ2）

- 勤務先/給与ルール/時間割/ユーザー更新APIへ `sync` 成功メタデータを追加した。
- `lib/query/mutation-toast.ts` を追加し、成功トーストの説明文を共通生成するようにした。
- シフト系と勤務先系の主要更新導線で、pending補足文言の判定を共通ヘルパー経由に統一した。

## 6. 追加実装（2026-06-02 フェーズ3）

- 月次シフト一覧とダッシュボードの削除導線で、`shifts.month` キャッシュを対象にした楽観的UIを追加した。
- 削除成功レスポンス待ちの間も対象シフトは即時に一覧/カレンダーから消え、失敗時だけ rollback する。
- シフト mutation 後の invalidation は `shifts` を優先し、給与集計・勤務先関連は一部導線でバックグラウンド invalidation に切り替えた。
- これにより、削除後の画面閉塞要因だった `reload()` 待ちを削除しつつ、最終整合は維持する。

## 7. 追加実装（2026-06-02 フェーズ4）

- `/my/shifts/new` と `/my/shifts/[id]/edit` は保存成功レスポンスから `MonthShift` を再構築し、戻り先遷移前に月次 cache を即時更新するようにした。
- `/my/shifts/confirm` は未確定一覧から対象カードを即時に除去し、右側の確定済み一覧へ `計算中` の provisional 行を追加する。
- シフト mutation 後の invalidation は待機させず background 実行に寄せ、画面遷移・一覧更新の体感待ちを削減した。

## 8. 追加実装（2026-06-02 フェーズ5）

- `/my/shifts/bulk` は保存成功レスポンスの `data[]` を使って、戻り先遷移前に月次 `shifts.month` cache を複数件まとめて即時更新するようにした。
- Google Calendar 同期失敗や再取得待ちがあっても、DB 保存済みなら画面遷移自体は止めず、関連 invalidation は background 実行に寄せた。
- これにより、一括登録後の `/my` 復帰時に一覧・カレンダーへ新規シフトが見えるまでの待ち時間を削減した。
