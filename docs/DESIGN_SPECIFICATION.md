# 1. システム概要

本システムは  
**個人用シフト・給与概算管理Webアプリ**である。

主な機能

- シフト登録・編集・削除
- 月間カレンダー表示
- 概算給与計算・集計
- Google Calendar同期

# 2. システム構成

**アーキテクチャレイヤー**

| レイヤー           | 技術スタック           | 説明                                     |
| ------------------ | ---------------------- | ---------------------------------------- |
| **フロントエンド** | Browser + React 19     | ユーザーブラウザで実行される UI レイヤー |

**認証**

- NextAuth.js を使用したメール認証
- 単一ユーザー対応

**通信フロー**

Browser (React UI) → Next.js Routes → Prisma ORM → Neon DB

外部連携: Neon ← → Google Calendar API

## 2.2 データ取得・キャッシュ戦略（2026-05 更新）

2026-05 のパフォーマンス改善で、読み取りキャッシュ責務を **Server Cache（Next.js Cache Components）** と **Client Cache（TanStack Query）** に整理した。

### サーバー側（Route Handler / Server Components）

- API レスポンスは `lib/api/cache-control.ts` を経由し、機密データを含むレスポンスを `private, no-store` 中心で返却する。
- `/my`・`/my/shifts/list` の初期月次シフト取得は `lib/shifts/month-shifts.ts` の共通 DAL で取得し、初回表示時の重複 fetch を避ける。
- 勤務先・給与ルール・時間割の参照系は `lib/cache/workplace-read-cache.ts` で `use cache` + `cacheTag` を利用し、Mutation 時は `lib/cache/revalidate.ts` の tag invalidation で再検証する。
- シフト系 Mutation（作成/更新/削除/確定/一括登録）でも user/workplace 領域の tag invalidation を実行し、Server Component 側の stale 表示を抑制する。

### クライアント側（React Query）

- App 全体を `QueryClientProvider` で包み、クライアント再表示時の読み取り状態を query cache で統一する。
- query key は `lib/query/query-keys.ts` で一元管理し、`shifts` / `payroll` / `workplaces` ドメインで衝突しないキー体系を維持する。
- `payroll` ドメインには `summary` / `details` / `previewBaseline` に加え `actual`（実給与編集）を含め、実給与更新後は給与サマリー・給与詳細・実給与編集を同時に再取得する。
- fetch は `lib/query/fetch-json.ts` で共通化し、エラー種別の正規化と UI メッセージ解決を統一する。
- 月次シフト・給与集計/詳細・勤務先管理一覧・シフト確定・各フォーム編集時の読み取りは TanStack Query を利用する。
- 単体シフト入力と一括入力の勤務先初期データは `/api/shifts/form-bootstrap` で一括取得し、勤務先一覧 + 選択勤務先詳細 + 給与ルール + 時間割セットをまとめて扱う。
- 月/年切替や再取得では `placeholderData` により直前データを維持し、置換型ローディングではなくオーバーレイ型ローディングを表示する。
- 初回表示は full spinner / skeleton を許容し、再取得時は `aria-busy=true` を付与した対象領域に `最新データを更新中...` オーバーレイを重ねる。
- Mutation 後の整合は `lib/query/invalidation.ts` のドメイン単位 invalidation で行い、手動 reload を原則廃止する。
- シフト削除導線は月次 `shifts.month` キャッシュを楽観的に更新し、DB削除成功前でも一覧・カレンダーから即時に消える体験を優先する。
- 単体シフトの登録・編集は保存成功レスポンスの `data` を使って `shifts.month` キャッシュへ即時反映し、遷移は再取得完了待ちでブロックしない。
- 一括登録も保存成功レスポンスの `data[]` を使って `shifts.month` キャッシュへ即時反映し、複数件登録後も `/my` 遷移を重い invalidation で待たせない。
- シフト確定ページは未確定一覧からの即時除去に加え、確定済み一覧へ `計算中` の provisional 行を追加して再取得までの空白をなくす。
- シフト更新後の再取得は `shifts` を優先し、給与集計・勤務先関連の invalidation は導線に応じてバックグラウンド化を許容する。
- シフト系の更新APIは `sync` オブジェクト（`status`/`pending`/`requiresCalendarSetup`/`requiresSignOut` など）を返し、DB更新成功と外部同期状態を分離して扱う。
- 勤務先/給与ルール/時間割/ユーザー更新APIも `sync` を返し、外部同期がない操作では `status=success` として扱う。
- クライアントは保存成功後の遷移を同期完了待ちでブロックせず、`sync.pending=true` の場合は「Google Calendar 同期はバックグラウンドで実行中」と明示する。
- Google Calendar 初期化後の既存シフト同期も同様にバックグラウンド実行とし、初期化画面の遷移を待たせない。
- 旧 `lib/client-cache/*` は廃止し、二重キャッシュを解消する。

### 初期表示と hydration

- Server Component で取得した初期データは `initialData` として query に注入する。
- `useMonthShifts` は `staleTime` を設定し、hydration 直後の即時 refetch によるローディング退行を抑制する。
- Google Calendar 予定参照 API（`/api/calendar/events`）は専用の短期メモリキャッシュを持ち、期限切れ時は stale fallback と警告表示を許容する（書き込み同期の正は DB）。

# 3. ドメインモデル

本システムは単一ユーザー専用である。各ドメインモデルの関連性は以下の通り。

**エンティティ関連性**

- **User** (ユーザー)
  - **Workplace** (勤務先) - 複数登録可能
    - **PayrollRule** (給与ルール) - 複数、時系列管理
    - **Timetable** (時間割) - 複数、塾のみ
    - **Shift** (シフト) - 複数
      - **ShiftLessonRange** (授業コマ範囲) - 複数、LESSON タイプのシフトのみ

## 3.1 User

ログインユーザー

**説明**: NextAuth.js でログイン管理されるユーザー。本システムは単一ユーザー対応のため、通常1件のみ存在する。

| 属性      | 型       | 説明                         |
| --------- | -------- | ---------------------------- |
| id        | UUID     | ユーザーの一意識別子         |
| email     | string   | メールアドレス（ログインID） |
| createdAt | datetime | アカウント作成日時           |

**制約条件**

- email は一意（unique）
- email は有効なメール形式

**使用例**

- email: user@example.com
- createdAt: 2026-01-15T08:30:00Z

---

## 3.2 Workplace

勤務先（複数登録可能）

**説明**: ユーザーが勤務する企業・塾など。シフトの根拠となる組織。

| 属性      | 型       | 説明                                       |
| --------- | -------- | ------------------------------------------ |
| id        | UUID     | 勤務先の一意識別子                         |
| name      | string   | 勤務先名（例：「コンビニA」「塾B」）       |
| color     | string   | UIに表示する色（HEX形式）                  |
| type      | enum     | 勤務先の種別（GENERAL または CRAM_SCHOOL） |
| createdAt | datetime | 勤務先登録日時                             |

**制約条件**

- name は空でない
- color は HEX カラーコード（#RRGGBB）形式
- 1ユーザーあたり複数勤務先可能
- 削除時は関連する Shift, PayrollRule も削除される（CASCADE）

**使用例**

- name: "コンビニA"、type: GENERAL、color: "#FF5733"
- name: "英語塾"、type: CRAM_SCHOOL、color: "#3366FF"

**type の説明**

- **GENERAL**: 通常の勤務先（コンビニ、飲食店など）
- **CRAM_SCHOOL**: 塾（Timetable と ShiftLessonRange を使用）

---

## 3.3 PayrollRule

給与ルール履歴（勤務先ごとの時給管理）

**説明**: 勤務先ごとの時給・割増ルールを時系列管理。同一勤務先で複数ルールが存在し、シフト日に該当する有効なルールを自動選択する。

**ルール選択ロジック**: `startDate ≤ shift.date < endDate` を満たすルール中、`startDate` が最新（最大）のものを選択。同一勤務先内では適用期間の重複を許さない。

| 属性                   | 型      | 説明                                          |
| ---------------------- | ------- | --------------------------------------------- |
| id                     | UUID    | ルールの一意識別子                            |
| workplaceId            | UUID    | 勤務先ID（外部キー）                          |
| startDate              | date    | ルール適用開始日                              |
| endDate                | date?   | ルール適用終了日（NULLの場合は現在有効）      |
| baseHourlyWage         | number  | 基本時給（円、小数2位まで。NORMAL/OTHER型用） |
| perLessonWage          | number? | コマ給（円、小数2位まで。LESSON型用）         |
| holidayAllowanceHourly | number  | 休日手当（円/時の加算手当。デフォルト0）      |
| nightPremiumRate       | number  | 深夜割増率（例：0.25 = 25%）                  |
| dailyOvertimeThreshold | number  | 1日の所定時間（超過分が残業対象）             |
| overtimePremiumRate    | number  | 残業割増率（将来拡張。例：0.25 = 25%）        |
| holidayType            | enum    | 休日判定タイプ                                |

**制約条件**

- startDate と endDate が重複しない（同一勤務先内で）
- endDate は NULLか startDate より後
- baseHourlyWage > 0
- perLessonWage > 0
- nightPremiumRate ≥ 0
- overtimePremiumRate ≥ 0
- dailyOvertimeThreshold > 0（時間、実数可）

**holidayType の説明**

- **NONE**: 休日判定なし（毎日同じ時給）
- **WEEKEND**: 土日を休日扱い
- **HOLIDAY**: 指定の祝日のみ休日扱い（将来拡張）
- **WEEKEND_HOLIDAY**: 土日と祝日を休日扱い

**使用例**

勤務先「コンビニA」:

- Rule 1: 2026-01-01 ～ 2026-03-31、baseHourlyWage=1100、nightPremiumRate=0.25
- Rule 2: 2026-04-01 ～ NULL、baseHourlyWage=1150、nightPremiumRate=0.25

勤務先「塾B」（CRAM_SCHOOL）:

- Rule 1: 2026-01-01 ～ 2026-03-31、perLessonWage=2000、nightPremiumRate=0.0
- Rule 2: 2026-04-01 ～ NULL、perLessonWage=2500、nightPremiumRate=0.0

## 3.3.1 ActualPayroll

実給与実績（勤務先ごとの支給月実績）

**説明**: 給与明細をもとに、勤務先ごとの支給月実績を保持する。シフト由来の概算給与は残しつつ、表示上は実績値を優先する。

| 属性             | 型       | 説明                                             |
| ---------------- | -------- | ------------------------------------------------ |
| id               | UUID     | 実給与レコードの一意識別子                       |
| workplaceId      | UUID     | 対象勤務先                                       |
| paymentMonth     | date     | 支給月（`YYYY-MM-01` として保持）                |
| taxableAmount    | decimal  | 課税対象額。給与・賞与など年収の壁判定対象を含む |
| nonTaxableAmount | decimal  | 非課税対象額。交通費など年収の壁判定対象外を含む |
| note             | string?  | 補足メモ。任意                                   |
| createdAt        | datetime | 登録日時                                         |
| updatedAt        | datetime | 更新日時                                         |

**制約条件**

- 単位は `workplaceId + paymentMonth` の一意制約とする。
- `taxableAmount`, `nonTaxableAmount` は 0 以上。
- 控除・手取り額は保持しない。
- 実績未登録はレコードなしで表現する。

## 3.4 Timetable

塾の時間割（CRAM_SCHOOL タイプの勤務先のみ使用）

**説明**: 塾の授業コマごとの開始・終了時刻を定義。通常期間と講習期間など複数のセットを管理可能。シフト登録（LESSON型）時に、選択したコマ範囲から`startTime/endTime`を自動計算する際に使用。

**用途**：

- LESSON型シフト時：「5限～7限」のように選択すると、該当コマの時刻を自動取得
- NORMAL型の場合も塾に登録可能（手入力、会議や事務作業など授業以外の業務を記録時に使用）

| 属性        | 型   | 説明                                                                    |
| ----------- | ---- | ----------------------------------------------------------------------- |
| id          | UUID | 時間割の一意識別子                                                      |
| workplaceId | UUID | 勤務先ID（CRAM_SCHOOL のみ、外部キー）                                  |
| type        | enum | 授業タイプ（NORMAL または INTENSIVE）                                   |
| period      | int  | コマ番号（1, 2, 3, ... など。typeごとに独立、例：通常期1限、講習期1限） |
| startTime   | time | コマの開始時刻（HH:MM）                                                 |
| endTime     | time | コマの終了時刻（HH:MM）                                                 |

**制約条件**

- workplaceId は type が CRAM_SCHOOL
- (workplaceId, type, period) は一意（type ごとに独立した period 番号体系を持つ）
- startTime と endTime は同時刻不可
- endTime < startTime の場合、翌日終了のコマとして扱う（例：23:50 ～ 翌00:30）

**type の説明**

- **NORMAL**: 通常期間の授業
- **INTENSIVE**: 講習期間（集中講座・特別授業等）

同一勤務先・同一period内で、両typeのコマを登録可能（時期に応じて使い分け）。

**使用例**

塾「英語塾」の時間割（通常期と講習期で時間帯が異なる場合の例）:

通常期（type=NORMAL）:

- Period 1: 16:30 ～ 17:30
- Period 2: 17:40 ～ 18:40
- Period 3: 18:50 ～ 19:50

講習期（type=INTENSIVE）:

- Period 1: 15:00 ～ 16:20
- Period 2: 16:30 ～ 17:50
- Period 3: 18:00 ～ 19:20

## 3.5 Shift

シフト（実際の勤務）

**説明**: ユーザーが登録する実際の勤務シフト。勤務先ごと・日ごとに複数登録可能。Google Calendar と同期。

**LESSON型の特徴**：

- ユーザーは「5限～7限」のようにコマ範囲を選択
- Timetableから該当コマの時刻を自動計算 → `startTime/endTime` に格納
- ShiftLessonRange で「どのコマ範囲か」を記録
- Google Calendar には自動計算された時刻で登録される

**NORMAL型（塾での会議・事務作業など授業以外の業務）の特徴**

- ユーザーが時刻を手入力
- 時給で給与計算される（PayrollRule の baseHourlyWage を使用）

| 属性          | 型       | 説明                                             |
| ------------- | -------- | ------------------------------------------------ |
| id            | UUID     | シフトの一意識別子                               |
| workplaceId   | UUID     | 勤務先ID（外部キー）                             |
| date          | date     | シフトの勤務開始日                               |
| startTime     | time     | シフト開始時刻（HH:MM）                          |
| endTime       | time     | シフト終了時刻（HH:MM）                          |
| breakMinutes  | int      | 休憩時間（分単位、デフォルト0）                  |
| shiftType     | enum     | シフトのタイプ（NORMAL, LESSON, OTHER）          |
| comment       | string?  | 表示・Google Calendar イベント名用の任意コメント |
| isConfirmed   | boolean  | シフト確定フラグ（デフォルト false）             |
| googleEventId | string?  | Google Calendar のイベントID（同期用）           |
| createdAt     | datetime | シフト登録日時                                   |

**制約条件**

- 同一勤務先の時間帯重複は、日跨ぎ解釈後の開始日時・終了日時で判定する（バリデーション段階で警告）
- startTime と endTime は同時刻不可
- endTime < startTime の場合、翌日終了のシフトとして扱う
- breakMinutes ≥ 0
- comment は任意。未入力・空文字・空白のみは NULL として保存
- comment は最大100文字、改行不可
- shiftType = LESSON の場合、ShiftLessonRange が1件以上存在
- shiftType = LESSON 以外の場合、ShiftLessonRange は存在しない

**勤務先表示名・イベント名**

- comment が NULL の場合: `勤務先名`
- comment がある場合: `勤務先名 (コメント)`
- 従来の塾タイプ NORMAL シフトに対する自動の ` (事務)` 表記は廃止する。事務として表示したい場合は comment に `事務` を入力する。

**shiftType の説明**

- **NORMAL**: 通常勤務（手入力。勤務先がCRAM_SCHOOLの場合は会議や事務作業など授業以外の業務も含む。時給で給与計算）
- **LESSON**: 塾の授業（CRAM_SCHOOL のみ。ShiftLessonRange で「何コマ目から何コマ目まで」を定義。時刻は Timetable から自動計算）
- **OTHER**: その他（自由入力）

**使用例**

- 2026-03-20、コンビニA、10:00 ～ 18:00、NORMAL、breakMinutes=60
- 2026-03-21、塾B、16:30 ～ 19:50、LESSON（Period 1～3）、breakMinutes=0

---

## 3.6 ShiftLessonRange

塾のコマ範囲（Shift.shiftType = LESSON の場合のみ）

**説明**: LESSON タイプのシフトが どのコマから どのコマまでをカバーするかを記録。Timetable のコマに対応する。複数コマの授業を1シフトで記録可能。

**注記**：`startPeriod ≤ endPeriod` の同じ `type`（NORMAL or INTENSIVE）のコマ範囲を指す。異なる `type` のコマを混在させることはできない。

| 属性        | 型   | 説明                                         |
| ----------- | ---- | -------------------------------------------- |
| id          | UUID | レコードの一意識別子                         |
| shiftId     | UUID | シフトID（外部キー）                         |
| startPeriod | int  | 開始コマ番号                                 |
| endPeriod   | int  | 終了コマ番号（startPeriodと同じ場合は1コマ） |

**制約条件**

- shiftId の Shift.shiftType = LESSON
- startPeriod ≤ endPeriod
- startPeriod, endPeriod は Timetable のコマの period に存在し、かつ同一 `type`（NORMAL or INTENSIVE）
- 1シフトあたり1件（複数コマの授業を1つのシフトで表現）

**使用例**

- Shift（2026-03-21、塾B、16:30 ～ 19:50、LESSON）
  → ShiftLessonRange: startPeriod=1, endPeriod=3

## UC-001: シフト登録

**目的**: 新しいシフトをDB に登録し、Google Calendar に同期する。

**前提条件**

- ユーザーがログイン状態
- 勤務先が最低1件登録済み
- （LESSON の場合）塾の時間割が設定済み

**フロー**

1. ユーザーがカレンダーから日付をクリック
2. シフト入力フォーム表示（勤務先、シフトタイプなど）
3. ユーザーが情報を入力
   - **NORMAL型の場合**：開始時刻・終了時刻を手入力
   - **LESSON型の場合**：コマ範囲（e.g. Period 1～3）を選択 → サーバー側で Timetable から時刻を自動計算
4. クライアント側でバリデーション（必須項目、時刻の妥当性など）
5. NORMAL/OTHER型で終了時刻 < 開始時刻の場合、翌日終了として扱う確認ダイアログを表示
6. バリデーション OK → POST /api/shifts（Shift + ShiftLessonRange を作成）
7. サーバー側で：
   - LESSON型の場合、Timetable から `startTime/endTime` を自動計算
   - Shift を INSERT → ShiftLessonRange を INSERT
8. DB 保存成功 → POST /api/calendar-sync にイベント作成を依頼（自動計算した時刻を使用）
9. Google Calendar API でイベント作成 → googleEventId を Shift に UPDATE
10. 成功レスポンス → 前画面へ戻る + 成功メッセージ表示

**エラーハンドリング**

- バリデーションエラー → UI でユーザーに指摘し、修正を促す
- Google Calendar API エラー → DB には保存済み、スタッシュして後で再同期（将来）

# 8. 給与計算仕様

## 8.1 対象・非対象

**対象**

- **NORMAL/OTHER型**: 基本給、休日手当（時間あたり加算）、深夜割増
- **LESSON型**: コマ給（perLessonWage）

**非対象（MVP では未対応）**

- 控除（税金、社会保険など）
- 週単位での残業計算
- 月60時間を超える残業（法定外残業）
- 通勤手当・その他手当

---

## 8.2 計算の基本フロー

1. 対象期間内のすべての Shift を取得
2. Shift ごとに該当する PayrollRule を時系列で確定
3. Shift ごとに以下を計算：
   - **NORMAL/OTHER型**: 実勤務時間、深夜帯勤務時間、休日時間を分類し、給与を計算
   - **LESSON型**: コマ数（endPeriod - startPeriod + 1）と perLessonWage を用いて給与を計算
4. 期間合計を集計

---

## 8.3 計算式

### 8.3.1 勤務時間の分類

**実勤務時間（H_total）**

```
H_total = (endTime - startTime - breakMinutes) / 60 [時間]
```

- endTime < startTime の場合、endTime は翌日の時刻として計算する。

**深夜帯勤務時間（H_night）**

- 22:00 ～ 05:00 の時間帯に該当する勤務時間（固定）
- 例：22:00 ～ 05:00 の場合
  - 22:00 ～ 24:00（2時間）と 00:00 ～ 05:00（5時間）の合計
  - 翌日にまたがる場合は合算

**基本時間（H_base）**

```
H_base = H_total - H_night
```

**休日時間（H_holiday）**

```
H_holiday = 休日判定に合致する日の場合 H_total、それ以外は 0
```

### 8.3.2 給与計算

**NORMAL/OTHER型のシフト（1シフト分）**

基本前提

- 休日判定は `holidayType` に基づく
- 深夜割増率は `nightPremiumRate`（0以上）
- 休日手当は `holidayAllowanceHourly`（円/時、0以上）

```
基本支給 = H_base × baseHourlyWage

深夜支給 = H_night × baseHourlyWage × (1 + nightPremiumRate)

休日支給 = H_holiday × holidayAllowanceHourly

シフト合計給 = round(基本支給) + round(深夜支給) + round(休日支給)
```

※ `overtimePremiumRate` は将来拡張用に保持するが、現時点のV2計算では使用しない。

**LESSON型のシフト（1シフト分）**

```
コマ数 = endPeriod - startPeriod + 1

シフト合計給 = コマ数 × perLessonWage
```

**補足**

- 休日時間は基本時間・深夜時間とは独立に扱う（休日夜勤では深夜支給と休日支給の両方が発生）
- 詳細なルールは実装時に業務ロジックで定義

---

## 8.4 具体例

### 例1：通常勤務（NORMAL シフト）

**条件**

- 勤務先：コンビニA（GENERAL）
- 日付：2026-03-20（金、平日）
- 時間：10:00 ～ 18:00
- 休憩：60分
- PayrollRule：baseHourlyWage=1100, nightPremiumRate=0.25, holidayAllowanceHourly=0

**計算**

```
H_total = (18:00 - 10:00 - 60) / 60 = 7 時間
H_night = 0（深夜帯なし）
H_base = 7 時間
H_holiday = 0 時間

基本支給 = 7 × 1100 = 7,700 円
深夜支給 = 0 円
休日支給 = 0 円

給与 = 7,700 円
```

### 例2：夜勤シフト（NORMAL シフト、深夜帯含む）

**条件**

- 勤務先：別のコンビニ
- 日付：2026-03-21（土、休日判定で WEEKEND）
- 時間：22:00 ～ 翌05:00
- 休憩：30分
- PayrollRule：baseHourlyWage=1100, holidayAllowanceHourly=100, nightPremiumRate=0.25, holidayType=WEEKEND

**計算**

```
H_total = (05:00 (次日) - 22:00 - 30) / 60 = 6.5 時間
H_night = 6.5 時間（全て深夜帯）
H_base = 0 時間
H_holiday = 6.5 時間

基本支給 = 0 × 1100 = 0 円
深夜支給 = 6.5 × 1100 × 1.25 = 8,937.5 円 → 8,938 円
休日支給 = 6.5 × 100 = 650 円

給与 = 8,938 + 650 = 9,588 円
```

### 例3：塾の授業（LESSON シフト）

**条件**

- 勤務先：塾B（CRAM_SCHOOL）
- 日付：2026-03-22（日）
- シフトタイプ：LESSON（Period 1～3、type=NORMAL）
- Timetable（通常期間）：
  - Period 1 (NORMAL): 16:30 ～ 17:30
  - Period 2 (NORMAL): 17:40 ～ 18:40
  - Period 3 (NORMAL): 18:50 ～ 19:50
- 自動計算：startTime=16:30、endTime=19:50、コマ数=3
- PayrollRule：perLessonWage=2000

**計算**

```
コマ数 = 3 - 1 + 1 = 3 コマ

給与 = 3 × 2000 = 6,000 円
```

### SCR_003: カレンダー

**目的**: 月間シフト一覧を視覚的に表示し、シフト登録・編集・削除のエントリーポイントを提供する。

**ヘッダー機能**

| 要素       | 仕様                                                     |
| ---------- | -------------------------------------------------------- |
| 月選択     | 前月・次月ボタン、または月ピッカー                       |
| 表示月     | 「2026年3月」など YYYY年M月 形式で表示                   |
| 当日マーク | 本日の日付の背後に薄めの塗りつぶし丸を配置（ハイライト） |

**カレンダー表示**

- **形式**: 月間カレンダー（日曜始まり）
- **日付セル**: 日付の数字を表示
- **空きセル**: 前月・次月の日付は薄く表示（クリック不可）

**シフト表示（新規仕様）**

​日付セルの下部に、その日に登録されているシフト数に応じて色付き丸を表示：

- **1件のシフト**: そのシフトの勤務先の色の丸（直径約6～8px）を1つ表示
- **複数件**: 各シフトの勤務先色に対応した丸を**横並び**に表示（最大5～6件、超過は「+N」テキスト）
- **色の由来**: 各丸の色は `Workplace.color` フィールドから取得
- **配置**: 日付の数字より下、セルの中央下部に配置

**例**

```
┌─────────────┐
│      5      │   ← 日付数字
│             │
│    ● ●     │   ← シフトが2件（勤務先Aとワークプレイス Bの色の丸）
└─────────────┘

┌─────────────┐
│      12     │
│             │
│  ● ● ● +2  │   ← シフトが5件以上（色の丸3つ + "+2" テキスト）
└─────────────┘
```

**インタラクション**

| アクション           | 動作                                                                                            |
| -------------------- | ----------------------------------------------------------------------------------------------- |
| **日付クリック**     | シフトがない場合 → SCR_004 (シフト入力) へ遷移                                                  |
| **シフト丸クリック** | その日のシフト一覧モーダルを表示 → シフト選択で編集/削除                                        |
| **マウスホバー**     | 丸上に勤務先表示名をツールチップ表示（2秒後に表示）。comment がある場合は `勤務先名 (コメント)` |

**シフト一覧モーダル**（日付クリック時の選択肢）

- **表示内容**: その日のシフト一覧（時刻、勤務先表示名、給与予想含む）。comment がある場合、勤務先表示名は `勤務先名 (コメント)`
- **アクション**:
  - シフト行クリック → SCR_005 (シフト編集) へ遷移
  - 「新規追加」ボタン → SCR_004 (シフト入力) へ遷移
  - 「削除」アイコン → SCR_006 (削除確認) へ遷移

**ローディング表示**

- 初回表示時は全体スピナーを表示してよい。
- 月切替や再取得時は、既存の月間カレンダーを維持したまま `最新データを更新中...` オーバーレイを重ねる。
- 再取得中のカレンダー領域には `aria-busy=true` を付与する。

### SCR_004: シフト入力

**目的**: 新規シフトをDB に登録し、Google Calendar に同期する。

**フォーム定義**

| フィールド   | 型     | 必須 | 仕様                                                                                                 |
| ------------ | ------ | ---- | ---------------------------------------------------------------------------------------------------- |
| 勤務先       | SELECT | ○    | 登録済み勤務先から選択; デフォルト = 前回選択した勤務先                                              |
| 日付         | DATE   | ○    | カレンダーピッカー or 手入力; デフォルト = クリック日                                                |
| シフトタイプ | RADIO  | ○    | NORMAL / LESSON / OTHER; デフォルト = NORMAL                                                         |
| 開始時刻     | TIME   | ◆    | NORMAL/OTHER型のみ表示; HH:MM 形式; バリデーション: 00:00 ～ 23:59                                   |
| 終了時刻     | TIME   | ◆    | NORMAL/OTHER型のみ表示; HH:MM 形式; 開始時刻と同時刻は不可。開始時刻より早い場合は翌日終了として扱う |
| 休憩時間     | NUMBER | ○    | 分単位; デフォルト = 0; 0 ～ 240                                                                     |
| コマ種別     | RADIO  | ◆    | LESSON型のみ表示; NORMAL（通常期）/ INTENSIVE（講習期）                                              |
| コマ範囲     | SELECT | ◆    | LESSON型のみ表示; Period 1～N; startPeriod ≤ endPeriod を確認                                        |
| コメント     | TEXT   | ー   | 任意; 最大100文字; 改行不可; 空文字・空白のみは NULL として保存                                      |

**説明**

- **◆ 条件付き必須**: シフトタイプに応じて表示・必須が変わる
  - NORMAL/OTHER型: `開始時刻`、`終了時刻` は必須。`コマ種別`、`コマ範囲` は非表示
  - LESSON型: `コマ種別`、`コマ範囲` は必須。`開始時刻`、`終了時刻` は非表示（Timetableから自動計算）
- コメント入力欄の下に Google Calendar のイベント名登録イメージをリアルタイム表示する。
  - comment 入力あり: `イベント名プレビュー「勤務先名 (コメント)」`
  - comment 未入力: `イベント名プレビュー「勤務先名」`
  - 勤務先未選択の場合は、プレビューを非表示にするか `勤務先を選択するとイベント名を確認できます` と表示する。

**バリデーション**

1. **必須チェック**: 条件付き必須項目を確認
2. **時刻バリデーション**:
   - NORMAL/OTHER型: 開始時刻と終了時刻は同時刻不可
   - NORMAL/OTHER型で終了時刻 < 開始時刻の場合、翌日終了として扱い、保存前に確認ダイアログを表示する
   - （警告）日跨ぎ解釈後の時間帯が既存シフトと重複している場合の警告
3. **コマ範囲**: LESSON型選択時、有効なコマが存在するか確認
4. **コメント**: 最大100文字、改行不可。空文字・空白のみは NULL

**日跨ぎ確認ダイアログ**

- 対象: NORMAL/OTHER型で終了時刻 < 開始時刻の場合
- 表示タイミング: 登録送信前
- 表示内容: 開始日時、終了日時、翌日終了として登録する旨
- アクション: `キャンセル` / `翌日終了として保存`

**SCR_005: シフト編集時の差分仕様**

- シフト編集フォームは SCR_004 と同じ時刻解釈を使用する。
- 編集後に終了時刻 < 開始時刻となる場合、保存前に日跨ぎ確認ダイアログを表示する。
- 既存シフトがすでに日跨ぎで、開始時刻・終了時刻を変更していない場合は確認を省略する。

**エラーメッセージ**

| エラーコード | メッセージ                                 |
| ------------ | ------------------------------------------ |
| ERR_001      | 〇〇は必須項目です                         |
| ERR_002      | 開始時刻と終了時刻は同じ時刻にできません   |
| ERR_003      | この日付にはすでにシフトが登録されています |
| ERR_004      | 塾の授業は時間割が登録されていません       |

- **レイアウト**: 各日付を独立したカード/セクション形式で縦並びに表示（チェックボックス + 日付 + 入力フィールド）
- **スクロール**: 複数日同時登録時は、選択日の数に応じてスクロール可能な高さ制限を設ける（例：最大高さ 400px～600px）
- **常時展開**: 完全に展開された状態で表示され、たためる機能は不要
- **デフォルト値の有効性**: デフォルト値を適用した後でも、各日付ごとに個別に値を編集可能
- **イベント名プレビュー**: 各日付のコメント入力欄の下に `イベント名プレビュー「勤務先名 (コメント)」` を表示し、個別コメント編集時に即時更新する。デフォルト値適用時は、各日付のプレビューも適用後のコメントで更新する

**ステップ4: 確定**

| ボタン     | 動作                                  |
| ---------- | ------------------------------------- |
| 登録       | 全選択日のシフトをDB登録 + 同期       |
| キャンセル | SCR_002（/my のカレンダー表示）へ戻る |

### 1. 目的

- 勤務先ごとに異なる「締日」「給料日」を扱えるようにする。
- `/my/summary` は「選択月に実際に入ってくる収入（支給月基準）」で集計する。
- 年収の壁を意識した判断のため、年内受取見込を明確に表示する。

---

### 2. 用語定義

- 締日:
  - 各勤務先の給与計算期間を区切る日付。
  - 設定値は `1..31` または `月末締め`。
- 給料日:
  - 各勤務先の支給日（日のみ）。設定値は `1..31`。
  - 実日付は対象月の日数に合わせて丸める（例: 31日指定で2月は末日）。
- 支給月:
  - SCR_007 でユーザーが選択した月（`YYYY-MM`）。
- 対象勤務期間:
  - 支給日の実日付から過去に遡って直近の締日を `endDate` とし、その1つ前の締日の翌日を `startDate` とする期間。
  - 期間は `startDate` と `endDate` の両端を含む。

---

### 3. ドメイン変更（Workplace）

#### 3.1 追加する設定値

- 勤務先ごとに以下を保持する。
  - `締日設定`: `DAY_OF_MONTH`（日付指定）または `END_OF_MONTH`（月末）
  - `締日`（日付指定時のみ）: `1..31`
  - `給料日`: `1..31`

#### 3.2 制約

- `給料日` は必須。
- `締日設定 = DAY_OF_MONTH` のとき `締日` は必須。
- `締日設定 = END_OF_MONTH` のとき `締日` は保持しない（または `null`）。
- 締日と給料日が同日となる設定は不可（バリデーションエラー）。

#### 3.3 既存データ移行方針

- 既存勤務先の初期値は以下で固定する。
  - 締日設定: `END_OF_MONTH`（月末締め）
  - 給料日: `25`

### 4. 集計仕様（SCR_007）

#### 4.1 期間選択

- 期間選択は「月選択のみ」。
- カスタム期間指定は廃止する。

#### 4.2 支給月の主集計

選択月 `M` に対して、勤務先ごとに以下を計算する。

1. `M` の支給日実日付 `paymentDate` を求める（`給料日` を月末丸め）。
2. `paymentDate` 以下で直近の締日を `periodEndDate` とする。
3. `periodEndDate` より1つ前の締日の翌日を `periodStartDate` とする。
4. その勤務先のシフト（未確定含む）を `periodStartDate..periodEndDate` で集計する。
5. 集計結果を勤務先別内訳として表示する。

全体サマリーは、勤務先別集計の総和とする。

#### 4.3 勤務先別内訳の表示列

| 列名     | 内容                               |
| -------- | ---------------------------------- |
| 勤務先名 | Workplace.name                     |
| 対象期間 | `periodStartDate 〜 periodEndDate` |
| 勤務時間 | 支給対象期間の実勤務時間           |
| 給与小計 | 支給対象期間の概算給与             |

- 「対象期間」は勤務先名の右に表示する。

#### 4.4 副表示（ラベル変更）

- 旧「当月累計」→ `年内受取累計（選択月まで）`
  - 定義: 選択年の1月〜選択月に支給される収入の合計。
- 旧「年間累計」→ `年間受取見込（1月〜12月）`
  - 定義: 選択年の1月〜12月に支給される収入の合計見込。
  - 目的: 年収の壁確認に使える値を提供する。

補足:

- いずれも「勤務日基準」ではなく「支給日基準」で集計する。
- 未来月は登録済みシフトのみを根拠に見込計算し、未登録分は 0 とする。

#### 4.5 確定済み支給額

- 「前月合計」は廃止する。
- 代わりに `確定済み支給額` を表示する。
  - 定義: 選択月の支給対象期間に含まれるシフトのうち、`isConfirmed = true` の概算給与合計。
  - 目的: 当月支給見込のうち、確定済みとして確度が高い金額を即時に把握できるようにする。

### 5. アルゴリズム詳細

#### 5.1 日付丸め

- `day` 指定が対象月の末日を超える場合は末日に丸める。
  - 例: 31日指定 + 2026-02 => 2026-02-28

#### 5.2 締日列挙

- `DAY_OF_MONTH(d)`:
  - 各月の締日は `min(d, 月末日)`。
- `END_OF_MONTH`:
  - 各月の締日は月末日。

#### 5.3 対象勤務期間の導出

- `paymentDate` を基準に、`paymentDate` 以下で最大の締日を `periodEndDate` とする。
- `periodEndDate` より前で最大の締日を `previousClosingDate` とする。
- `periodStartDate = previousClosingDate + 1日`。

#### 5.4 例

- 例1: 締日=15日, 給料日=25日, 支給月=2026-04
  - `paymentDate = 2026-04-25`
  - `periodEndDate = 2026-04-15`
  - `periodStartDate = 2026-03-16`
  - 対象期間: `2026-03-16 〜 2026-04-15`

- 例2: 締日=月末, 給料日=20日, 支給月=2026-04
  - `paymentDate = 2026-04-20`
  - `periodEndDate = 2026-03-31`
  - `periodStartDate = 2026-03-01`
  - 対象期間: `2026-03-01 〜 2026-03-31`

### 5. 給与内訳・計算ロジックの表示定義

本章は「表示上の定義」。計算エンジン自体は既存ロジックを利用する。

#### 5.1 時間

- `勤務時間(H_total)` = `(endTime - startTime - breakMinutes) / 60`
- `基本勤務時間(H_base)`:
  - 休日判定が `false` のシフトでは `H_total`
  - 休日判定が `true` のシフトでは `0`
- `休日勤務時間(H_holiday)`:
  - 休日判定が `true` のシフトでは `H_total`
  - 休日判定が `false` のシフトでは `0`
- `深夜勤務時間(H_night)` = 設定された深夜帯との重複時間
- `残業時間(H_overtime)` = `max(0, H_total - dailyOvertimeThreshold)`
- 時間表示ルール:
  - テーブル表示は `h:mm` 形式（例: `7:30`）
  - 計算内部は従来どおり小数時間で扱う
  - `H_night` は `H_base` / `H_holiday` と重複しうる（加算内訳のため）

#### 5.2 時給の選択

- 休日判定が `true` の場合:
  - `holidayHourlyWage` があればそれを使用
  - なければ `baseHourlyWage` を使用
- 休日判定が `false` の場合:
  - `baseHourlyWage` を使用

#### 5.3 金額内訳（1シフト）

- `基本勤務金額`:
  - 平日シフト: `H_total × 適用時給`
  - 休日シフト: `0`
- `休日勤務金額`:
  - 休日シフト: `H_total × 適用時給`
  - 平日シフト: `0`
- `深夜勤務金額`:
  - `H_night × 適用時給 × (nightMultiplier - 1)`
- `残業金額`:
  - `H_overtime × 適用時給 × overtimeMultiplier`
- `シフト合計`:
  - `基本勤務金額 + 休日勤務金額 + 深夜勤務金額 + 残業金額`

注記:

- 現行実装準拠で、`残業金額` は「残業時間分の加算」として上式で表示する。
- LESSON 型シフトも現行実装では時給計算で集計する。

#### 5.4 月合計・年合計

- 月合計 = 対象シフトの `シフト合計` 総和
- 年合計 = 対象年1月〜12月の月合計総和

#### 5.5 金額表示ルール

- 金額の単位表記は `¥` を使用する。
- 表記例: `¥44,720`

# 15. シフト登録時の翌月支給額プレビュー仕様（2026-05-10 統合）

本章は、シフト登録時に未保存シフトを反映した概算支給額を即時確認するための仕様である。

詳細な実装計画は `docs/archive/feature-plans/2026-05-10_shift-payroll-preview-plan.md` を参照する。

対象:

- SCR_004（シフト入力）
- SCR_014（シフト一括登録）

非対象:

- SCR_005（シフト編集）
- SCR_015（シフト確定）
- Google Calendar 側予定の取り込み・逆同期

---

### 1. 目的

- シフト登録前に、入力中の未保存シフトを含めた支給月別の概算支給額を確認できるようにする。
- ユーザーが支給見込を見ながらシフト入力・一括入力を調整できるようにする。
- 一括登録で複数支給月に渡る場合でも、支給月ごとの影響額を把握できるようにする。

---

### 2. 表示単位

表示単位は「支給月」とする。

- 「翌月支給額」は単純な翌カレンダー月ではなく、13章の給与締日・給料日仕様に基づく `支給月` として扱う。
- 勤務先ごとの締日・給料日により、同じ勤務日でも属する支給月が異なる場合がある。
- 入力中シフトが複数支給月に分かれる場合、該当する全支給月を昇順で表示する。

例:

- 月末締め / 25日払いの勤務先では、`2026-06-10` のシフトは `2026-07` 支給に含まれる。
- 15日締め / 25日払いの勤務先では、`2026-06-10` のシフトは `2026-06` 支給、`2026-06-20` のシフトは `2026-07` 支給に含まれる。

---

### 3. 表示内容

支給月ごとに以下を表示する。

| 項目           | 内容                                             |
| -------------- | ------------------------------------------------ |
| 支給月         | `YYYY年M月支給`                                  |
| 現在の支給見込 | DB 保存済みシフトのみで計算した支給額            |
| 追加予定額     | 入力中の未保存シフトのみで計算した支給額         |
| 登録後見込     | `現在の支給見込 + 追加予定額`                    |
| 対象シフト数   | その支給月へ反映される未保存シフト件数           |
| 計算状態       | 未入力、不足情報あり、給与ルール不足、計算済み等 |

表示ルール:

- 金額は `¥12,345` 形式で表示する。
- 追加予定額は `+¥3,200` のように差分であることが分かる形式にする。
- `登録後見込` を最も強調する。
- 計算中表示は入力操作をブロックしない。

---

### 4. 金額計算

各支給月のプレビューは以下で計算する。

```text
登録後見込 = 現在の支給見込 + 入力中シフトの追加予定額
```

現在の支給見込:

- DB 保存済みシフトだけを対象にする。
- 13章の支給月基準で集計する。
- 未確定シフトも含める。
- Google Calendar 側の予定は含めない。

追加予定額:

- 入力中フォームの未保存シフトだけを対象にする。
- クライアント側で即時計算する。
- 給与計算式は8章および `docs/PAYROLL_CALCULATION_SPEC_V2_20260430.md` に従う。
- プレビュー専用の独自計算式は作らない。
- LESSON型では、選択したコマ範囲の時間割から `startTime/endTime` と `breakMinutes` を算出する。
  - コマ間ギャップを `breakMinutes` として扱う。
  - 算出ロジックはシフト保存時（`/api/shifts`）と同一の共通関数を利用する。

---

### 5. 入力不備時の扱い

入力途中で計算できないシフトがあっても、プレビュー全体は破綻させない。

| 状態                   | 表示方針                                               |
| ---------------------- | ------------------------------------------------------ |
| 勤務先未選択           | `勤務先を選択すると支給額を確認できます`               |
| 日付未入力             | `日付を入力すると支給月を判定できます`                 |
| 時刻未入力             | `開始時刻と終了時刻を入力すると追加予定額を計算します` |
| 同時刻エラー           | フォームバリデーションを優先し、金額は表示しない       |
| 給与ルール不足         | `この日付に適用できる給与ルールがありません`           |
| 時間割不足             | `塾の時間割が登録されていません`                       |
| 一括登録の一部行が不備 | 有効行だけを合算し、未計算件数を補足表示する           |

一括登録で一部行に不備がある場合:

- 有効行の追加予定額は表示する。
- `未計算: N件` を表示する。
- 登録ボタン押下時の既存バリデーションは維持する。

---

### 6. UI仕様

PC:

- 画面右下にフローティング表示する。
- 登録フォームの送信ボタンや主要入力欄を覆わない位置に配置する。
- 一括登録で支給月が多い場合、プレビュー内を縦スクロール可能にする。

スマホ:

- 画面下部にフローティング表示する。
- 初期状態はコンパクト表示にする。
- タップで詳細を展開できるようにする。
- 送信ボタンと重なる場合は、フォーム下部に十分な余白を確保する。

アクセシビリティ:

- フローティング領域はラベル付きの補助領域として扱う。
- 金額更新の読み上げは必要最小限にする。
- モバイルの展開ボタンは `aria-expanded` を設定する。
- キーボード操作で展開・折りたたみできるようにする。

### 7. API方針

プレビュー専用の軽量 API を追加する。

候補:

```text
GET /api/payroll/preview-baseline?months=YYYY-MM,YYYY-MM
```

責務:

- 認証済みユーザーの支給月ごとの既存支給見込を返す。
- DB 保存済みシフトだけを対象にする。
- Google Calendar 側の予定は含めない。
- プレビューに必要な最小データだけを返す。

レスポンス主要項目:

- `month`
- `totalWage`
- `byWorkplace[].workplaceId`
- `byWorkplace[].wage`
- `byWorkplace[].periodStartDate`
- `byWorkplace[].periodEndDate`

### 8. パフォーマンス方針

- 登録画面ロード時に年間全月の支給額は取得しない。
- 入力中シフトから該当支給月を判定し、必要な支給月だけ baseline を取得する。
- baseline は TanStack Query でキャッシュする。
- 入力変更のたびに API を呼ばない。
- 未保存シフト分の追加予定額はクライアントで計算する。
- 給与ルール、給与サイクル、時間割はフォーム既存取得データを可能な限り再利用する。
- 一括登録では給与ルール・給与サイクル・時間割を `Map` 化し、入力行ごとの過剰な探索を避ける。

# 16. 実給与登録・実績優先表示仕様（2026-06-18）

対象:

- SCR_007（給与集計）
- SCR_016A / SCR_016B（給与詳細）
- SCR_017（実給与編集）
- `ActualPayroll` ドメイン

### 1. 目的

- 給与明細で確定した実際の支給額を、支給月単位で登録できるようにする。
- 年収の壁判定に使うため、`課税対象` と `非課税対象` を分けて保持する。
- 給与サマリーと給与詳細では、実績がある箇所は実績値を優先表示し、概算値は比較用に残す。

### 2. 登録単位と入力項目

- 登録単位は `勤務先 × 支給月` とする。
- 支給月は `YYYY-MM` で選択し、DB では月初日 `YYYY-MM-01` として保持する。
- 入力項目:
  - `課税対象額`
  - `非課税対象額`
  - `メモ`（任意）
- `実合計 = 課税対象額 + 非課税対象額`
- 両金額とも 0 以上。両方未入力かつメモ空の行は「未登録」として扱い、保存時はレコード削除対象とする。

### 3. 画面仕様

#### 3.1 SCR_017 実給与編集

- ルート: `/my/payroll/actual`
- サイドバーの `給与管理` 配下から遷移できる。
- 月選択で対象支給月を切り替える。
- 勤務先一覧には以下を表示する。
  - 勤務先名
  - 対象期間
  - 概算額
  - 課税対象額
  - 非課税対象額
  - 実合計
  - 差額（実合計 - 概算額）
  - メモ
- 保存は月単位一括保存とする。

#### 3.2 SCR_007 給与集計

- 総額カードは `実績支給額` を主表示とする。
- `実績支給額` は、勤務先ごとに `実給与があれば実績、なければ概算` を採用して合算する。
- `年内受取累計` と `年間受取見込` も、各月の `実績支給額` 合計を用いる。
- `概算給与` は比較用に併記する。

#### 3.3 SCR_016 給与詳細

- 月毎表示:
  - ヘッダーは `実績支給額` を最優先で表示する。
  - `実績支給額` カード内に `課税/非課税内訳（課税/非課税/合計）` を含め、`概算支給合計` を併記する。
  - 勤務先別には `表示金額` と `概算金額` を併記し、計算根拠の時間内訳・式は概算由来のまま維持する。
- 勤務先毎表示:
  - 年間主表示は、12か月分の `実績支給額` 合計を使う。
  - 月次表も `実績支給額` を優先し、`概算額` を比較列として残す。

### 4. API方針

- 新規 API:
  - `GET /api/payroll/actual?month=YYYY-MM`
  - `PUT /api/payroll/actual?month=YYYY-MM`
- `GET` は対象月の全勤務先について、概算値・対象期間・既存実給与値を返す。
- `PUT` は対象月の複数勤務先入力を受け取り、`workplaceId + paymentMonth` で upsert する。
- 既存 API の返却値には、表示用の `display` 概念を追加する。
  - `summary`: 月合計・累計・年合計に実績優先値を含める
  - `details/monthly`: 勤務先別・ヘッダー向けの実績優先値を含める
  - `details/workplace-yearly`: 月次行・年次カード向けの実績優先値を含める

### 5. 表示優先ルール

- 勤務先月次の主表示金額:
  - 実給与登録あり -> `実合計`
  - 実給与登録なし -> `概算額`
- 月合計・年合計・累計:
  - 各勤務先/各月の主表示金額を積み上げた値を正とする。
- 概算由来の勤務時間、深夜時間、休日時間、計算式は引き続き表示する。

### 6. 受け入れ条件

1. `/my/payroll/actual` から支給月別の実給与を登録できる。
2. 課税対象額と非課税対象額を分けて保存できる。
3. `給与サマリー` の主表示額と累計/年計が実績優先になる。
4. `給与詳細（月毎）` で実績値と概算値の両方を確認できる。
5. `給与詳細（勤務先毎）` の年合計は実績優先表示になる。
