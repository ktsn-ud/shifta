# シフト登録時の翌月支給額プレビュー 実装計画

作成日: 2026-05-10

## 1. 目的

シフト登録時に、登録予定のシフトを反映した翌月以降の概算支給額を即時プレビューできるようにする。

ユーザーはシフト入力中に「このシフトを入れると、次に受け取る給与がいくらになりそうか」を確認しながら調整できる。特に一括登録では複数日の入力が支給額へ与える影響を把握しにくいため、右下または下部に常時参照できるフローティング表示を追加する。

データの正はアプリ DB とし、Google Calendar 側の予定はこのプレビュー計算に含めない。

## 2. 要件整理

- 対象画面:
  - `SCR_004: シフト入力`（`/my/shifts/new`）
  - `SCR_014: シフト一括登録`（`/my/shifts/bulk`, `/my/bulk`）
- 非対象:
  - `SCR_005: シフト編集`
  - `SCR_015: シフト確定`
  - Google Calendar 予定の取り込み・逆同期
- 登録しようとしているシフトを含めた支給額合計をプレビューする。
- 表示単位は「支給月」単位とする。
- 登録予定シフトが複数の支給月に渡る場合、該当する全支給月をそれぞれ表示する。
- PC では画面右下付近にフローティング表示する。
- スマホでは画面下部にフローティング表示する。
- 入力変更に対して即時に再計算する。
- 登録前プレビューのため、DB へ保存しない。
- 実装はテスト駆動開発で進める。
- 各実装タスクでは、先に失敗するテストを書き、その後に実装してテストを通す。

## 3. 決定事項

- 「翌月支給額」は、単純な翌カレンダー月ではなく、勤務先ごとの締日・給料日に基づく `支給月` 基準で扱う。
- 一括登録で一部行に入力不備がある場合、有効行だけを計算し、未計算件数を表示する。
- Baseline 取得は、プレビュー専用の軽量 API を追加する。
- スマホ表示は、下部固定 + 初期コンパクト + タップで詳細展開を基本とする。
- `SCR_005: シフト編集` への適用は初期スコープ外とし、必要になった場合に差分設計を別途行う。

## 4. 既存仕様との整合

### 4.1 支給月基準

既存の給与集計仕様では、勤務先ごとに `締日` と `給料日` が異なり、`/my/summary` は「支給月基準」で集計する。

この機能も同じ支給月基準を採用する。

例:

- 勤務先A: 月末締め / 25日払い
- `2026-06-10` のシフトは `2026-07` 支給に含まれる。
- 勤務先B: 15日締め / 25日払い
- `2026-06-10` のシフトは `2026-06` 支給に含まれる。
- `2026-06-20` のシフトは `2026-07` 支給に含まれる。

ユーザー要件の「翌月支給額」は、実装上は単純な翌カレンダー月ではなく、勤務先の給与サイクルに基づく `支給月` として扱う。

### 4.2 給与計算ロジック

計算ロジックは既存の `lib/payroll/*` と同じ意味を維持する。

- NORMAL/OTHER 型:
  - 基本勤務
  - 深夜勤務
  - 休日手当
  - 残業割増は現行 V2 仕様どおり計算対象外
- LESSON 型:
  - 既存実装の扱いに合わせる
  - 実装時に `calculateLessonShiftWage` と `summarizeByPeriod` の差分がないか確認する

注記:

- `docs/PAYROLL_CALCULATION_SPEC_V2_20260430.md` を正本とする。
- プレビュー専用の別計算式を作らず、サーバー側集計とクライアント側プレビューで同じ計算関数を共有できる形へ寄せる。

## 5. 仕様案

### 5.1 表示する値

フローティングプレビューには、支給月ごとに以下を表示する。

| 項目           | 内容                                             |
| -------------- | ------------------------------------------------ |
| 支給月         | `YYYY年M月支給`                                  |
| 現在の支給見込 | DB に保存済みの既存シフトだけで計算した合計      |
| 追加予定額     | 入力中で未保存のシフトだけで計算した合計         |
| 登録後見込     | `現在の支給見込 + 追加予定額`                    |
| 対象シフト数   | その支給月へ反映される未保存シフト件数           |
| 計算状態       | 未入力、不足情報あり、給与ルール不足、計算済み等 |

初期表示では、入力中シフトが有効でない場合もカードを表示する。

- 単体登録: `シフト情報を入力すると支給額を確認できます`
- 一括登録: `日付とシフト情報を入力すると支給額を確認できます`

### 5.2 表示対象の支給月

単体登録:

- 入力中シフトが有効に計算できる場合、そのシフトが属する支給月を表示する。
- 日付・勤務先・給与ルールが未確定の場合は金額を出さず、案内表示にする。

一括登録:

- 選択済みで、登録対象として有効な行だけをプレビュー対象にする。
- 複数支給月へ分かれる場合、支給月ごとのカードを複数表示する。
- 表示順は支給月の昇順とする。

### 5.3 金額の考え方

各支給月のプレビューは以下で計算する。

```text
登録後見込 = 現在の支給見込 + 入力中シフトの追加予定額
```

現在の支給見込:

- 登録画面ロード時、またはプレビュー対象の支給月が増えた時点でサーバーから取得する。
- 既存 DB のシフトを支給月基準で集計した合計。
- 未確定シフトも含める。これは `/my/summary` の主集計と同じ考え方にする。

追加予定額:

- 入力中のフォーム状態からクライアントで即時計算する。
- DB 未保存のため、Google Calendar 同期状態は考慮しない。
- 一括登録では、有効な行ごとに計算して支給月別に合算する。

### 5.4 入力不備時の扱い

入力中のシフトがまだ計算できない場合、プレビュー全体をエラーにしない。

| 状態                   | 表示方針                                               |
| ---------------------- | ------------------------------------------------------ |
| 勤務先未選択           | `勤務先を選択すると支給額を確認できます`               |
| 日付未入力             | `日付を入力すると支給月を判定できます`                 |
| 時刻未入力             | `開始時刻と終了時刻を入力すると追加予定額を計算します` |
| 同時刻エラー           | フォームバリデーションを優先し、金額は表示しない       |
| 給与ルール不足         | `この日付に適用できる給与ルールがありません`           |
| 時間割不足             | `塾の時間割が登録されていません`                       |
| 一括登録の一部行が不備 | 有効行だけ合算し、不備件数を補足表示する               |

一括登録で一部行だけ不備がある場合:

- 有効行の追加予定額は表示する。
- `未計算: N件` を表示する。
- 登録ボタン押下時の既存バリデーションはそのまま維持する。

### 5.5 UI仕様

PC:

- 画面右下に固定表示する。
- 登録フォームの送信ボタンや主要入力欄を覆わない位置に置く。
- 幅は `320px` から `380px` 程度を想定する。
- 一括登録で支給月が多い場合、カード内部を縦スクロール可能にする。

スマホ:

- 画面下部に固定表示する。
- 初期状態はコンパクト表示にする。
- タップで詳細を展開できる形を推奨する。
- 送信ボタンと重なる場合は、フォーム下部に十分な `padding-bottom` を追加する。

共通:

- 金額は `¥12,345` 形式で表示する。
- `追加予定額` は差分であることが分かるように `+¥3,200` のように表示する。
- `登録後見込` を最も強調する。
- 計算中のローディングはスケルトンまたは小さいスピナーに留め、入力操作をブロックしない。

### 5.6 アクセシビリティ

- フローティング領域は `aside` または意味のあるラベル付き領域として扱う。
- 更新される金額は過度に読み上げが発生しないよう、必要最小限の `aria-live="polite"` を検討する。
- モバイルの展開ボタンは `aria-expanded` を設定する。
- キーボード操作で展開・折りたたみできるようにする。

## 6. 推奨アーキテクチャ

### 6.1 全体方針

登録画面ロード時に、すべての支給月の集計を先読みするのではなく、プレビュー対象になった支給月だけを取得・キャッシュする。

理由:

- 一括登録で複数月に渡る可能性はあるが、通常は1から2支給月程度に収まる。
- 全勤務先・全年月の支給額を事前取得すると初期表示が重くなる。
- 入力中の即時性は、DB 集計よりも未保存シフトの差分計算に影響される。

推奨フロー:

1. 登録画面ロード時に勤務先・給与ルール・時間割など、フォームに必要なデータを取得する。
2. 入力中シフトから該当支給月をクライアントで判定する。
3. 該当支給月の既存支給見込が query cache にない場合だけ API で取得する。
4. 未保存シフト分はクライアントで即時計算する。
5. `既存支給見込 + 未保存シフト分` を表示する。

### 6.2 データ取得

新規 API を追加する。

```text
GET /api/payroll/preview-baseline?months=YYYY-MM,YYYY-MM
```

レスポンス案:

```ts
type PayrollPreviewBaselineResponse = {
  data: {
    months: {
      month: string;
      totalWage: number;
      byWorkplace: {
        workplaceId: string;
        wage: number;
        periodStartDate: string;
        periodEndDate: string;
      }[];
    }[];
  };
};
```

代替案:

- 既存の `/api/payroll/summary?month=YYYY-MM` を複数回呼ぶ。

新規 API を採用する。

理由:

- 一括登録では複数支給月が同時に必要になる。
- `summary` は画面表示用の副表示や詳細情報を含む可能性があり、プレビュー用途には過剰。
- preview baseline は `totalWage` と必要最小限の内訳だけに絞れる。

### 6.3 クライアント計算

クライアントで使うため、給与計算の純粋関数を `lib/payroll/preview` または既存 `lib/payroll` 配下に整理する。

必要な入力:

- 入力中シフト
- 勤務先
- 給与ルール一覧
- 時間割セット
- 給与サイクル設定

必要な出力:

```ts
type ShiftPayrollPreviewItem = {
  temporaryId: string;
  paymentMonth: string;
  wage: number | null;
  status: "ready" | "incomplete" | "missing-rule" | "invalid";
  message?: string;
};
```

設計上の注意:

- React コンポーネント内に給与計算を直接書かない。
- `useMemo` で入力状態からプレビュー結果を派生させる。
- `useEffect` で派生 state を再セットしない。
- 給与ルール探索は勤務先IDごとの `Map` を作り、入力行ごとの線形探索を避ける。
- 一括登録の行数が増えても、各入力ごとに全行を過剰再計算しないようにする。

### 6.4 TanStack Query

既存のクライアントキャッシュ方針に合わせ、TanStack Query を利用する。

query key 案:

```ts
queryKeys.payroll.previewBaseline(months);
```

方針:

- `months` はソートして key 化する。
- `enabled` は対象支給月が1件以上ある場合のみ `true`。
- `staleTime` は短めに設定する。例: 30秒から2分。
- シフト登録成功後は既存の `invalidateAfterShiftMutation` にプレビュー baseline も含める。

### 6.5 Server / Client Component 境界

Next.js 16 の App Router では、ページは Server Component としてデータ取得し、入力操作が必要なフォーム部分だけ Client Component として扱う。

既存フォームは Client Component なので、今回も以下の方針にする。

- API ルート・DB アクセス・認証確認はサーバー側。
- 入力中の差分計算・フローティング UI はクライアント側。
- Client Component へ渡す props はシリアライズ可能な最小データにする。
- サーバー専用 DB ロジックを Client Component へ import しない。

## 9. パフォーマンス方針

### 9.1 推奨

- 既存 DB 集計は支給月単位でオンデマンド取得する。
- 入力中シフトの追加予定額はクライアントで計算する。
- 給与ルール・給与サイクル・時間割はフォーム既存取得データを可能な限り再利用する。
- 支給月別 baseline は TanStack Query でキャッシュする。
- 一括登録の入力行は `Map` と `useMemo` で支給月別にまとめる。

### 9.2 避けるべきこと

- 入力変更のたびに API を呼ぶ。
- 登録画面ロード時に年間全月の支給額を取得する。
- React state に派生結果を保存し、`useEffect` で再計算結果を setState する。
- コンポーネント内に給与計算式を直接実装する。
- サーバー専用の Prisma 依存モジュールを Client Component へ import する。

### 9.3 再計算コスト対策

一括登録で行数が多い場合の対策:

- 給与ルールは `workplaceId` ごとに事前に sort 済み配列へ変換する。
- 支給月判定に必要な給与サイクルは勤務先ごとに `Map` 化する。
- 時間割コマは `timetableSetId + period` の key で `Map` 化する。
- プレビュー表示は支給月単位の集計結果だけを render し、行ごとの詳細表示は初期スコープに含めない。

## 10. 受け入れ条件

1. `/my/shifts/new` で、計算可能な入力状態になると支給月別の概算支給額プレビューが表示される。
2. `/my/shifts/new` で、勤務先・日付・時刻・休憩時間・コマ範囲を変更するとプレビューが即時更新される。
3. `/my/shifts/bulk` で、選択日と入力内容に応じて支給月別の概算支給額プレビューが表示される。
4. 一括登録で複数支給月に渡る場合、支給月ごとに `現在の支給見込`、`追加予定額`、`登録後見込` が表示される。
5. 入力不備がある場合、プレビューは画面を壊さず案内または未計算件数を表示する。
6. PC では右下、スマホでは下部にフローティング表示される。
7. 入力変更ごとに API が連続発火しない。
8. シフト登録成功後、関連する給与集計キャッシュが無効化される。
9. Google Calendar 側の予定は計算に含まれない。
10. 実装タスクごとに、該当テストを先に作成してから実装している。

## 12. 決定事項の補足

### Q1. 「翌月支給額」の定義

決定:

- 勤務先ごとの締日・給料日に基づく `支給月` として扱う。

理由:

- 既存の `/my/summary` と同じ支給月基準に揃えるため。

### Q2. 単体編集画面への適用

決定:

- 初期スコープでは `登録` のみ対象とし、編集画面は含めない。

理由:

- 編集では既存シフトの差し引きが必要になり、`変更前の支給月・金額` と `変更後の支給月・金額` の差分設計が別途必要になるため。

### Q3. 一括登録で不備行がある場合の表示

決定:

- 有効行だけ計算し、未計算件数を表示する。

理由:

- 一括登録中の即時性を優先し、入力途中でも有効行の影響を把握できるようにするため。

### Q4. Baseline API の形

決定:

- プレビュー専用の軽量 API を追加する。

理由:

- 一括登録で複数支給月をまとめて取得し、既存 summary より小さいレスポンスにできるため。

### Q5. スマホ表示

決定:

- 下部固定 + 初期コンパクト + タップで詳細展開。

理由:

- フォーム入力領域を圧迫せず、必要な時だけ詳細を確認できるようにするため。
