# 日跨ぎシフト確認フロー 実装計画

作成日: 2026-05-02

## 1. 目的

終了時刻が開始時刻より早いシフトを、翌日終了のシフトとして登録できるようにする。

例: `2026-05-02 18:00 - 01:00` は、`2026-05-02 18:00` から `2026-05-03 01:00` までの勤務として扱う。

現状は一部画面で `開始時刻 < 終了時刻` を必須としているため、夜勤や深夜まで続く勤務が「開始時刻は終了時刻より前にしてください」として弾かれる。給与計算と Google Calendar 同期はすでに `endTime <= startTime` を翌日扱いする設計に近いため、画面・APIバリデーション・重複判定・仕様書を同じ考え方に揃える。

## 2. 決定事項

- `Shift.date` は勤務開始日として扱う。
- `startTime` と `endTime` は時刻のみを保持する。
- `endTime < startTime` の場合、終了日は `date + 1日` と解釈する。
- `endTime === startTime` は禁止する。
- 24時間以上の勤務、複数日をまたぐ勤務は MVP 対象外とする。
- DB スキーマ変更は行わない。
- Google Calendar 側で編集された内容はアプリへ逆同期しない。

## 3. ユーザー確認仕様

### 3.1 単体登録

対象:

- `SCR_004: シフト入力`
- `/my/shifts/new`

NORMAL シフトで `endTime < startTime` の場合、送信前に確認ダイアログを表示する。

表示内容:

- タイトル: `このシフトは日付をまたぎます`
- 本文:
  - 開始日時
  - 終了日時
  - `終了時刻が開始時刻より早いため、翌日終了として登録します。この内容で保存しますか？`
- ボタン:
  - `キャンセル`
  - `翌日終了として保存`

確認後に既存の登録 API へ送信する。送信 payload は既存形式を維持する。

### 3.2 単体編集

対象:

- `SCR_005: シフト編集`
- `/my/shifts/[id]/edit`

編集後の値が `endTime < startTime` になる場合、保存前に確認ダイアログを表示する。

ただし、既存シフトがすでに日跨ぎで、開始時刻・終了時刻を変更していない場合は確認を省略する。コメントなど時刻以外の編集だけで毎回確認が出ることを避けるため。

確認後に既存の更新 API へ送信する。送信 payload は既存形式を維持する。

### 3.3 シフト確定

対象:

- `SCR_015: シフト確定`
- `/my/shifts/confirm`

未確定シフトカードで `endTime < startTime` の状態で確定する場合、確定 API 送信前に確認ダイアログを表示する。

既存シフトがすでに日跨ぎで、開始時刻・終了時刻を変更していない場合でも、確定操作はユーザーが勤務実績を確認する操作なので、日跨ぎ解釈を明示してから確定する。

確認後に既存の確定 API へ送信する。

### 3.4 一括登録

対象:

- `SCR_014: シフト一括登録`
- `/my/shifts/bulk`
- `/my/bulk`

一括登録時、登録対象の中に `endTime < startTime` の NORMAL シフトが1件以上ある場合、API 送信前にまとめ確認ダイアログを表示する。

表示内容:

- タイトル: `翌日終了として登録されるシフトがあります`
- 本文:
  - 対象件数
  - 日付ごとの解釈一覧
- 一覧項目:
  - 対象日
  - 入力された時間帯
  - 解釈後の開始日時
  - 解釈後の終了日時
- ボタン:
  - `戻って修正`
  - `まとめて翌日終了として登録`

LESSON シフトは時間割から時刻が自動算出されるため、同じ判定ロジックで日跨ぎになった場合は一覧に含める。ただし初期実装では、画面上で時刻を直接編集できる NORMAL シフトの確認を優先する。

## 4. 時刻解釈仕様

共通ヘルパーで以下を扱う。

```ts
type ShiftTimeInterpretation = {
  isOvernight: boolean;
  startDateTimeLabel: string;
  endDateTimeLabel: string;
  displayTimeRange: string;
};
```

想定関数:

- `isOvernightShift(startTime, endTime)`
- `isSameTimeShift(startTime, endTime)`
- `getShiftEndDate(date, startTime, endTime)`
- `formatShiftDateTimeInterpretation(date, startTime, endTime)`
- `toComparableShiftRange(date, startTime, endTime)`

`endTime === startTime` は日跨ぎとして扱わず、バリデーションエラーにする。

## 5. 重複判定仕様

現状のフォーム側重複判定は同日内の分単位比較に依存しているため、日跨ぎシフトでは正しく判定できない。

修正方針:

- 比較用に開始日時・終了日時を絶対分へ正規化する。
- `endTime < startTime` の場合は終了側に1日分を加算する。
- 日跨ぎシフトとの重複を検知するため、候補日の前日・当日・翌日の既存シフトを取得対象に含める。
- 表示上は引き続き警告扱いとし、既存の「同日のシフトが重複している場合の警告」方針を維持する。

例:

| シフトA             | シフトB           | 判定   |
| ------------------- | ----------------- | ------ |
| 5/2 18:00 - 翌01:00 | 5/2 23:00 - 23:30 | 重複   |
| 5/2 18:00 - 翌01:00 | 5/3 00:30 - 02:00 | 重複   |
| 5/2 18:00 - 翌01:00 | 5/3 01:00 - 03:00 | 非重複 |
| 5/2 22:00 - 翌05:00 | 5/3 04:30 - 06:00 | 重複   |
| 5/2 10:00 - 18:00   | 5/2 18:00 - 22:00 | 非重複 |

## 6. 影響範囲

### 6.1 ドキュメント

- `docs/DESIGN_SPECIFICATION.md`
  - `Shift.date` を勤務開始日として明記する。
  - `startTime < endTime` を `startTime !== endTime` に変更する。
  - `endTime < startTime` は翌日終了として扱うことを明記する。
  - `SCR_004`, `SCR_005`, `SCR_014`, `SCR_015` の確認フローを追記する。
  - ERR_002 の文言を同時刻禁止に合わせて見直す。

### 6.2 共通ロジック

- `lib/api/date-time.ts` または `lib/shifts/time.ts`
  - 日跨ぎ判定・終了日解釈・表示用フォーマット・重複比較用レンジの共通関数を追加する。

### 6.3 単体登録・編集

- `components/shifts/ShiftForm.tsx`
  - `toMinutes(startTime) >= toMinutes(endTime)` のエラーを、同時刻のみエラーに変更する。
  - 日跨ぎ確認ダイアログを追加する。
  - 編集時は時刻変更有無を見て確認要否を判定する。
  - 重複判定を日跨ぎ対応の比較へ変更する。

### 6.4 シフト確定

- `components/shifts/ConfirmShiftCard.tsx`
  - 同時刻のみエラーに変更する。
  - 日跨ぎ確認ダイアログを追加する。
- `app/api/shifts/[id]/confirm/route.ts`
  - `toMinutes(startTime) >= toMinutes(endTime)` のエラーを、同時刻のみエラーに変更する。
  - サーバー側でも同時刻は禁止する。

### 6.5 一括登録

- `components/shifts/BulkShiftForm.tsx`
  - 各行の時刻バリデーションを同時刻のみエラーに変更する。
  - 日跨ぎ対象行の一覧を生成する。
  - API 送信前にまとめ確認ダイアログを表示する。
  - 確認後の再送信で二重確認にならないように、確認済み状態を明示的に管理する。
- `app/api/shifts/bulk/route.ts`
  - 既存の `buildShiftData` 経由で同時刻禁止を担保する。
  - 必要に応じて LESSON 算出時刻の同時刻禁止テストを追加する。

### 6.6 表示

以下の時間帯表示では、日跨ぎの場合に `翌` を付ける。

- ダッシュボードのシフト一覧モーダル
- シフト一覧
- シフト確定画面
- 給与明細系のシフト行

例:

- `18:00 - 翌01:00`
- `22:00 - 翌05:00`

## 7. 実装順序

1. 設計書更新
   - `docs/DESIGN_SPECIFICATION.md` の Shift 制約、画面仕様、エラー文言を更新する。
2. 共通時刻ヘルパー追加
   - 日跨ぎ判定、同時刻判定、表示、比較用レンジを共通化する。
3. 単体登録・編集対応
   - バリデーション変更、確認ダイアログ、重複判定修正を行う。
4. シフト確定対応
   - クライアントと API のバリデーションを変更し、確認ダイアログを追加する。
5. 一括登録対応
   - 行バリデーションを変更し、まとめ確認ダイアログを追加する。
6. 表示整備
   - 日跨ぎ時間帯を `翌HH:MM` 表記に統一する。
7. テスト追加・更新
   - 単体登録、編集、確定、一括登録、重複判定、給与計算、Google Calendar 同期の回帰を確認する。

## 8. テスト観点

### 8.1 単体登録・編集

- `18:00 - 01:00` を入力すると確認ダイアログが表示される。
- 確認後に payload が既存形式のまま送信される。
- `18:00 - 18:00` はエラーになる。
- `18:00 - 18:01` は確認なしで保存できる。
- 既存の日跨ぎシフトをコメントだけ編集しても確認が出ない。
- 既存の同日内シフトを日跨ぎへ変更すると確認が出る。

### 8.2 シフト確定

- `22:00 - 05:00` の未確定シフトを確定すると確認ダイアログが表示される。
- 確認後に `PATCH /api/shifts/[id]/confirm` が実行される。
- 同時刻はクライアントと API の両方でエラーになる。

### 8.3 一括登録

- 登録対象に日跨ぎシフトがない場合は確認なしで送信される。
- 登録対象に日跨ぎシフトがある場合は一覧確認が表示される。
- 一覧には対象日、入力時間、解釈後の開始日時・終了日時が表示される。
- `戻って修正` で送信されない。
- `まとめて翌日終了として登録` で1回だけ送信される。
- 複数日のうち日跨ぎ対象だけが一覧に出る。

### 8.4 重複判定

- 当日内の重複を検知する。
- 日跨ぎシフトと当日夜の重複を検知する。
- 日跨ぎシフトと翌日未明の重複を検知する。
- 終了時刻と開始時刻が一致する隣接シフトは非重複とする。

### 8.5 給与計算・Google Calendar

- `22:00 - 05:00` の給与計算が既存テストどおり通る。
- Google Calendar イベントの終了日時が翌日になる。
- 日跨ぎでないシフトの Calendar 同期挙動が変わらない。

## 9. 検証コマンド

コード実装時はタスク単位で以下を実行する。

```bash
pnpm exec tsc --noEmit
pnpm lint
pnpm test
pnpm format
```

この計画書のみを追加するコミットでは、ドキュメント変更のみのため `pnpm format` を実行する。

## 10. 非対象

- DB スキーマ変更。
- `endDate` または `endDateOffset` の追加。
- 24時間以上または複数日連続のシフト対応。
- Google Calendar からアプリ DB への逆同期。
- 休憩時間が勤務時間を超えるケースの仕様変更。
