# [issues-5] Google Calendar 同期 400 BAD REQUEST の予防策実装

## 概要

Google Calendar 同期時に 400 BAD REQUEST が発生し、再ログインと権限再確認で復旧した事象の再発を防ぐ。
以下のそれぞれ、コミットを分けて順番に実装する。

- fix: 同期前に OAuth トークン状態を検証し、期限切れ前にリフレッシュする。
  - リフレッシュ成功時に新しい refresh token が返る場合は保存値を更新する（ローテーション対応）。
  - `invalid_grant` 発生時は無限リトライせず、再認証が必要な状態として扱う。
- feat: 必要スコープの検証を追加し、スコープ不足時は同期を実行しない。
  - Google Calendar 同期に必要なスコープを定数化し、事前チェックを行う。
  - スコープ不足時は「再ログイン（再同意）が必要」であることをユーザーに明示する。
- fix: 400 系エラーの内訳を判定し、再試行ポリシーを明確化する。
  - `invalid_grant` / `insufficientPermissions` は再認証導線へ遷移する。
  - `429` / `5xx` は指数バックオフで再試行し、`4xx` のリクエスト不正は即時失敗にする。
- chore: Google API エラーの構造化ログと監視指標を追加する。
  - `reason` / `message` / 対象ユーザー / 対象 shiftId を記録し、再発傾向を追えるようにする。
  - 同一 reason の連続発生を検知できるよう、集計しやすい形式で出力する。
